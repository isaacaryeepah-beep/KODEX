API Documentation

DIKLY API

A read-only REST API for integrating your organization's Dikly data with HR tools, dashboards, and reporting systems.

Getting a key

API keys are issued by your organization's Dikly admin, from Admin Portal → Administration → API Access. Each key belongs to exactly one organization and is scoped to a specific set of data — an admin can create a key for you with only the scopes your integration needs.

Keys are shown in full only once, at creation. Dikly never stores or displays the full key again, so if it's lost, your admin will need to revoke it and issue a new one.

Quick start

Every request is authenticated with an X-API-Key header — no OAuth flow, no separate token exchange.

curl https://dikly.sbs/api/v1/ping \
  -H "X-API-Key: dk_live_your_key_here"

A working key returns your organization's name and the scopes it carries:

{
  "ok": true,
  "company": "Acme Ltd",
  "mode": "corporate",
  "keyName": "HR system integration",
  "scopes": ["read:attendance", "read:employees"]
}

Authentication

Send your key in the X-API-Key header on every request. There is no request signing, no expiry on the key itself, and no per-request timestamp — the key is the credential.

  • Base URL: https://dikly.sbs/api/v1
  • All responses are JSON, including errors.
  • A key only ever sees the one organization it was created for — there is no way to request another organization's data with a valid key, regardless of any parameter in the request.

Scopes

Every key carries an explicit whitelist of scopes, set by your admin at creation. An endpoint call without the required scope returns 403 naming exactly which scope is missing.

ScopeGrants
read:attendanceDaily clock-in/out records, hours worked, overtime
read:employeesEmployee directory (name, email, department, role)
read:leavesLeave requests and their statuses
read:shiftsShift definitions (name, hours, grace period)
Phase 1 of the API is read-only by design — there are no write scopes yet. This keeps the blast radius of a leaked key strictly limited to reading data, never changing it.

Pagination

List endpoints accept ?limit= (default 50, max 200) and ?offset=. Every list response has the same shape:

{
  "data": [ ... ],
  "pagination": { "total": 214, "limit": 50, "offset": 0 }
}

Errors

Errors are JSON with a machine-readable error code and a human-readable message.

StatuserrorMeaning
400bad_request / corporate_onlyMalformed input (e.g. an invalid date), or the endpoint doesn't apply to an academic-mode organization.
401missing_api_key / invalid_api_key / revoked_api_keyNo key sent, key not recognised, or key has been revoked.
402subscription_expiredThe organization's Dikly subscription/trial has ended — API access pauses along with the rest of the account.
403missing_scopeThe key doesn't have the scope this endpoint requires.
429rate_limitedToo many requests — see rate limits below. A Retry-After header (seconds) is included.
500server_errorSomething went wrong on Dikly's side. Safe to retry.

Rate limits

120 requests per minute, per key. The limit is tracked independently for each key, so one integration hitting its limit never affects another key on the same organization.

Endpoints

GET/ping

No scope required — verify a key is valid and see which scopes it carries. Useful as a first call when wiring up a new integration.

{
  "ok": true,
  "company": "Acme Ltd",
  "mode": "corporate",
  "keyName": "HR system integration",
  "scopes": ["read:attendance"]
}
GET/employees
read:employees

Query params: role, active (true/false), limit, offset

{
  "data": [
    {
      "id": "64f...",
      "name": "Dorothy Cobbinah",
      "email": "dorothy@acme.com",
      "employeeId": "DIK-EMP-0001",
      "department": "Operations",
      "role": "employee",
      "active": true,
      "createdAt": "2026-01-14T09:02:00.000Z"
    }
  ],
  "pagination": { "total": 42, "limit": 50, "offset": 0 }
}
GET/attendance
read:attendance

Corporate-mode organizations only. Query params: from, to (YYYY-MM-DD, inclusive), employeeId, limit, offset

{
  "data": [
    {
      "id": "64f...",
      "date": "2026-07-01",
      "employee": { "id": "...", "name": "Dorothy Cobbinah", "employeeId": "DIK-EMP-0001", "email": "dorothy@acme.com" },
      "shift": { "name": "Morning", "startTime": "08:00", "endTime": "17:00" },
      "status": "late",
      "clockIn": { "time": "2026-07-01T08:47:00.000Z", "method": "web", "late": true, "lateMinutes": 47, "note": "Traffic" },
      "clockOut": { "time": "2026-07-01T18:30:00.000Z", "method": "web", "earlyLeaveMinutes": 0, "note": "Project deadline" },
      "hoursWorked": 9.7,
      "overtimeHours": 0.7
    }
  ],
  "pagination": { "total": 1, "limit": 50, "offset": 0 }
}
GET/leaves
read:leaves

Corporate-mode organizations only. Query params: status (pending/approved/rejected), employeeId, limit, offset

{
  "data": [
    {
      "id": "64f...",
      "employee": { "id": "...", "name": "Dorothy Cobbinah", "employeeId": "DIK-EMP-0001", "email": "dorothy@acme.com" },
      "type": "annual",
      "startDate": "2026-08-01T00:00:00.000Z",
      "endDate": "2026-08-05T00:00:00.000Z",

  "pagination": { "total": 1, "limit": 50, "offset": 0 }

GET/shifts
read:shifts

Corporate-mode organizations only. No query params — returns every shift defined for the organization.

{
  "data": [
    { "id": "64f...", "name": "Morning", "startTime": "08:00", "endTime": "17:00", "gracePeriodMinutes": 15, "days": ["monday","tuesday","wednesday","thursday","friday"] }
  ],
  "pagination": { "total": 1, "limit": 1, "offset": 0 }
}

Data we don't expose

The API is deliberately limited to workforce/attendance data. It never returns pay amounts, currency, payment/subscription records, or anything from Dikly's billing system (Paystack) — that data doesn't leave Dikly through this API in any form.

Questions

If your key isn't behaving as documented here, or you need a scope this API doesn't yet support, contact support@dikly.sbs.
